    The general reply to this question is - I have not looked at the specific approach - that while you always have to read in everything, the potentially HUGE difference (as huge as the size of the data file) is if a) you have to read the entire file into memory, or b) you read chunks, process them, then read the next chunk. What is worse is the complexity of the algorithm, reading everything in and doing in-memory processing on the entire data set all at once is more simple.

    Karl SchultzCommunity TA· 6 days ago 
    Vlad,

    For reading large files, one approach would be to read the file in chunks, using read.csv() with the skip and nrows parameters.  You can filter each chunk for the date range you want.  Then use rbind() to append the results to a data frame.  This approach lets you use all the features of read.csv(). Since each chunk is a data frame, you can easily filter it.   And this should be pretty easy to implement.

    As you suggest, you can also use readLines(), grep, etc., to filter the data on a pure text stream.  But this only works if your text pattern matching is pretty straightforward. 

    Another tip for this project is to come up with a way to read the data faster.  I think we agree that most machines can probably read the entire file, but it will be slow and it will take a long time to test, revise, test, ... your code.  Look into using save() to write your filtered (subsetted) data to an R data file, and use load() to load it.  You can test for the existence of the R data file to decide if you can just load the R data file (faster), or load the original file, make the subset, and save the R data file.  That way, you only pay the price of the full load once.

    Karl SchultzCommunity TA· 5 days ago 

    The topic of reading large files is a bit of a side issue, but here is some code for one approach:

            data <- read.csv(datafile,
                         nrow = 1)
        colClasses <- lapply(data, class)
        chunkSize <- 100000
        currLine <- 1
        repeat {
            data2 <- read.csv(datafile,
                              colClasses = colClasses,
                              col.names = names(data),
                              nrow = chunkSize,
                              skip = currLine)
            currLine <- currLine + chunkSize
            rowsRead <- nrow(data2)
            # subset data2 here
            data <- rbind(data, data2)
            if (rowsRead != chunkSize) {
                break
            }
        }

    I removed some of the parameters in read.csv() that one might want for the assignment, such as sep. If you add them back, be sure to add them to both calls to read.csv(). You would also add code to subset or filter the data2 data frame to pick only the dates you want, where I put the comment.

    I tested by reading the file this way, and also with a single call to read.csv(). I then used the identical() function to compare the two data frames.

    Note that the rbind operation will get slower and slower as the file loads because it needs to copy the existing data the entire time. This also increases the memory requirement while it is running. You can also save some space by tossing out columns you know you won't need.

    I used the general idea of this approach and was able to get it to work.  I used read.table, but noticed that each iteration was taking longer and longer.  Dawned on me that the skip parameter still required those rows to be read in (get the end-of-line flag?).  Using open to establish and keep the connection to the file open meant that each read.table call started at the pointer from the previous read.table call.  Therefore no skip argument required and each iteration was the same duration.  It was actually pretty fast.

    dfile = file("filename")
    open(dfile, "rt")  ## establish the connection to the file

    readRows = 100000  ## chunk size I used
    currentLine = 1  ## not used except to provide feedback on progress

    repeat {
          dfepower <- read.table(dfile,  ...., nrows = readRows)  ## no skip argument
          currentLine = currentLine + readRows
          
          print(currentLine)  ## just to keep track of progress
          
          ...whatever you want to do

          if (nrow(dfepower) < readRows){
                break
          }
    }

    close(dfile)  ## close the connection to the file

    Gonzalo Rolón MorinelliSignature Track· 6 hours ago 
    From data.table's fread help page about it's input parameter:


    "Either the file name to read (containing no \n character), a shell command that preprocesses the file (e.g. fread("grep blah filename")) or the input itself as a string (containing at least one \n), see examples. In both cases, a length 1 character string. A filename input is passed through path.expand for convenience and may be a URL starting http:// or file://."

    That means that you can send the grep command directly as the input parameter and it would load to R only the contents of the temp file created with the grep's output (if you want to double-check that, simply look for a non-existent string sequence in the file and fread will raise a File Empty error with the temp file's location)

    I guess you can't know beforehand exactly how to filter the desired data, but it's useful to load a little chunk (like, say, first 5 lines or so) to peep the data's structure first and, once you've recognised it, create the grep filter to get the data. Or, if it's well documented, simply look at the README/Codebook files to see which data may be useful for filtering purposes.

    On our case, given that the Date fields are the ones at the beginning of line, it won't be much trouble filtering the desired data out.

    Also, as read loads a data.table instead of data.frame, it's quite fast.
